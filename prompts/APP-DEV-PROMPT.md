# 지시서 — 「기울어진 주사위」 앱 구현 착수

당신은 이 저장소의 앱을 구현한다. **설계는 이미 `APPDESIGNPROMPT.md`에 완결돼 있다 — 그 문서가 유일한 스펙이다.** 당신의 일은 새 설계가 아니라 그 스펙의 충실한 구현이다.

## 0. 먼저 읽을 것
1. `APPDESIGNPROMPT.md` **전체**(§0–§10) — 스펙. 특히 §6 기술 명세·§8 수용 기준·§9 마일스톤.
2. `data/worldview.json` — 신념도 장부(빌드 타임에 읽어 카드 뱃지 생성). 진실 원천.
3. `app-content/cards.json` — 전자책 카드 콘텐츠(별도 세션 산출, `prompts/EBOOK-CARDS-PROMPT.md`). 없으면 콘텐츠 파이프라인 골격만 먼저 만들고 목업 카드로 진행.
4. `EXPERIMENTS.md`·`PROTOCOL.md`·`prereg/` — 실행 탭 실험 A·B·C·안전 규칙 원천.

## 1. 스택·구조 (§6)
- **Flutter**(iOS+Android) 또는 React Native. 웹은 마케팅 랜딩+무료 미리보기.
- 하단 **2탭**([세계관] 전자책 / [실행] 실험 기록) + 프로필. IA는 §1.
- 백엔드: 인증·**단건 IAP(29,000원) 영수증 검증**·콘텐츠 CDN·암호화 백업·동의 관리·주간 리포트 스케줄러·집계 파이프라인·소유자 대시보드 API(§5.5).

## 2. 절대 지켜야 할 것 (협상 불가 — 위반 = 앱 정체성 파괴 or 위법)
- **정직성 표기**: 경험 주장 카드에 신념도 뱃지(빌드 타임에 `worldview.json`에서 읽음·하드코딩 금지), 기능 열은 "틀 채택 평가" 라벨. 금지 문구 린트 CI(카드+리포트+LLM 출력). §2.2·§10.
- **안전 게이트(§3.2)**: 무료·상시 접근. 자동 중단 규칙(기분·수면·기능 2영역 동시 악화 → 즉시 중단+상담 연결) **사용자가 끌 수 없음**. 안전 판정은 **반드시 규칙 기반**(LLM 위임 금지, §4.3 섹션 3).
- **데이터 4계층(§5)**: 계층 0(기분·수면) **온디바이스만·서버 미전송**. 계층 1 변환은 **온디바이스**(원문 기기 밖 금지). 계층 2·3 연구 동의는 별도 옵트인, **거부해도 앱·리포트 정상**. 결제 ≠ 연구 동의. 위기 사용자 데이터 전 계층 제외.
- **리포트 규칙표(§4.3)**: 국면 인지 규칙으로 생성, 신념도·한계 병기, 결과 약속·진단 금지. LLM 선택 계층은 서사만, 규칙 문장의 신념도·금지 문구 불변(출력 린트).
- **판정 외주(§3.3)**: 실행 C 시작 시 기준 수치 잠금, 66일 후 규칙 기반 자동 판정.

## 3. 마일스톤 (§9 그대로)
- **M1(3주)**: 콘텐츠 파이프라인 + 카드 뷰어(장 0–1) + 목차 + 커버리지 CI.
- **M2(4주)**: 전 장 변환 + 신념도/이원 회계 표기 + 페이월/단건 IAP + 프로필.
- **M3(4주)**: 실행 탭(실행 C 전체 + A·B) + 안전 게이트/자동 중단 + 로컬 기록.
- **M4(3주)**: 주간 리포트 엔진(규칙 기반) + 백엔드 백업 + 4계층 동의·투명성 화면.
- **M5(2주)**: 집계 파이프라인 + 소유자 대시보드(§5.5) + 선택적 LLM API 연동.
- **M6(2주)**: 접근성·법 고지·분석·스토어 심사 → 출시.

각 마일스톤 끝에 **§8 수용 기준**의 해당 항목을 QA로 확인하고 통과 여부를 보고하라.

## 4. 작업 방식
- 마일스톤 단위로 PR. 각 PR에 무엇을 구현했고 어떤 수용 기준을 통과했는지 명시.
- 스펙이 모호하면 **임의 결정하지 말고** 질문. 특히 정직성·안전·데이터 동의는 절대 임의 완화 금지.
- 신념도·실험 설계·안전 규칙의 값을 바꾸지 말 것 — 저장소 원천(worldview.json·EXPERIMENTS.md·prereg)이 진실이다.
- LLM 관련 구현 시 안전·판정은 규칙 기반이 최종이라는 원칙을 코드로 강제(LLM 출력을 신뢰 경로에 두지 말 것).

## 5. 첫 턴에 할 일
1. `APPDESIGNPROMPT.md` 전체 정독 후, M1 범위의 구현 계획을 제시.
2. 스택 확정(Flutter 권장) + 리포지토리 구조 제안.
3. 콘텐츠 파이프라인(원천 → 카드 JSON → 렌더) 골격부터 착수.
